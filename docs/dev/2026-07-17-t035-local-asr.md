# T035 接入本地 ASR 与显式云端降级

## 任务目标

使用 whisper-rs 接入本地 Whisper 转写、模型管理和显式云端降级；本地模式默认不发送云端请求，并记录实际使用的 Provider。

## 实际改动

- 引入当前 `whisper-rs` 0.16.0，使用 whisper.cpp 官方 `ggml-base-q5_1.bin` 模型。
- 新增本地模型管理：状态、下载进度、临时文件保护、大小校验、验证加载和删除。
- 模型保存在应用数据目录 `models` 下，不进入仓库；下载地址使用 whisper.cpp 官方 Hugging Face 仓库。
- 新增本地 WAV 解码、声道合并、f32 归一化和 16 kHz 线性重采样。
- 模型上下文按 canonical path 缓存，每次转写创建独立 Whisper state。
- 安装 whisper-rs 官方日志 hook，在未配置 log backend 时丢弃 whisper.cpp 内部输出，避免记录识别文本。
- ASR Provider 增加 `local`；本地模型缺失或推理失败时默认直接报错，不发起云端请求。
- `allow_cloud_fallback` 默认关闭；用户开启后才按 `fallback_asr_provider` 使用智谱或 OpenAI 配置。
- ASR 结果增加实际 Provider、模型和是否云端降级，历史记录新增 `used_asr_fallback` 快照。
- 设置页增加本地模型下载、验证、删除、进度和云端降级配置。
- 就绪检查按本地模型是否存在判断，不要求 API Key；正常录音、历史重新转写和就绪检查共享统一 Provider 配置。
- 本地首版只支持 WAV；应用录音天然兼容，MP3 上传继续使用云端 Provider。
- README、需求、方案、本地 ASR 计划和任务跟踪均同步实际行为。

## 为什么这么做

云端 ASR 存在网络、隐私和服务可用性依赖。本地模式需要在不破坏现有智谱/OpenAI Provider 的前提下提供离线能力，并明确防止“选择本地模式后静默上传云端”。

旧计划固定了过时的 whisper-rs 版本和自动降级行为。实施前重新核验 crates.io、whisper-rs 和 whisper.cpp 官方模型仓库，采用当前 0.16.0 API 和官方量化模型。

## 执行复盘

### 遇到的问题

1. 当前机器缺少 whisper.cpp 构建所需 CMake。
2. whisper.cpp debug 构建默认会输出 token 和识别文本，违反敏感日志约束。
3. 现有 ASR 结果只包含文本，无法记录本地失败后实际使用的降级 Provider。
4. 录音设备采样率通常不是 Whisper 要求的 16 kHz。
5. 旧本地 ASR 计划中的 whisper-rs 版本和模型体积已过时。

### 解决流程

1. 安装 CMake，并在 README 中加入源码构建依赖。
2. 使用 whisper-rs `install_logging_hooks` 接管 whisper.cpp/ggml 日志；项目未启用日志 backend，因此内容被安全丢弃。
3. 扩展 ASR 结果和历史快照，记录实际 Provider、模型和降级状态。
4. 在 Rust 中实现多声道合并和线性重采样到 16 kHz。
5. 通过 crates.io、whisper-rs 源码和 whisper.cpp 官方模型仓库重新确认 0.16.0 API 与 Q5_1 模型。

### 经验总结

- 隐私模式的降级必须是显式选择，不能把可用性优化变成隐式数据上传。
- 本地模型集成不仅是推理函数，还包括下载原子性、缓存、日志、采样率和失败说明。
- 真实模型 smoke test能够发现纯 mock 测试无法覆盖的本地构建和运行问题。

## 未完成事项

- Windows whisper.cpp 编译和真实本地推理需由 Windows CI 与桌面环境确认。
- 本地首版仅支持 WAV，不处理 MP3 解码。
- 当前使用 CPU 推理，尚未启用 CUDA、Vulkan、Metal 等硬件加速 feature。
- 本地文本润色和流式转写不在本任务范围。

## 后续建议

收集本地与云端 ASR 的实际耗时和错误率，再决定是否增加硬件加速、本地文本模型或流式转写。
