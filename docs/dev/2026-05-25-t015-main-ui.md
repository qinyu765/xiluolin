# T015 实现主界面

> **归档说明：** 本文记录特定开发阶段的背景与决策，其中的 MVP、demo、比赛或旧分支流程表述仅用于保留历史，不代表 XiLuoLin 当前的开源项目定位与协作方式。当前信息请以根目录 `README.md`、`CONTRIBUTING.md` 和 `docs/roadmap.md` 为准。

## 任务目标

实现语音输入主界面，包含：
1. 当前人格选择器
2. 录音按钮（支持点击开始/停止）
3. 录音状态展示（录音中、处理中、完成、失败）
4. 原始识别文本展示区
5. 整理后文本展示区
6. 复制按钮
7. 输出按钮（调用 output_text 命令）

4. **改造主界面卡片**：

### 后端改动

1. **新增 Tauri 命令** (src-tauri/src/pipeline.rs):

2. **注册新命令** (src-tauri/src/lib.rs):
   - 在 `invoke_handler` 中注册 `pipeline::process_recording_file`

## 为什么这么做

### 架构选择

1. **前端不直接读取文件**：
   - Tauri v2 的文件系统插件需要额外安装和配置
   - 在后端读取文件更简单、更安全
   - 避免前端处理大型二进制数据

2. **复用现有流程**：
   - `process_recording_file` 内部调用 `process_voice_input`
   - 保持 ASR → 文本整理 → 历史保存的统一流程
   - 减少代码重复

3. **录音时长实时显示**：
   - 提供视觉反馈，让用户知道录音正在进行
   - 使用 100ms 间隔更新，平衡性能和流畅度

### UI 设计

1. **大型圆形录音按钮**：
   - 符合语音输入应用的常见设计模式
   - 录音中变为红色方块，视觉上清晰表示"停止"操作
   - 尺寸 64x64px，易于点击

2. **人格选择前置**：
   - 在录音前选择人格，避免录音后还需要选择
   - 显示人格描述，帮助用户快速理解

3. **保留文件上传**：
   - 作为测试和备选方案
   - 用分隔线和"或上传音频文件"文字明确区分

4. **输出按钮**：
   - 调用 `output_text` 命令，支持键盘注入或剪贴板粘贴
   - 与"复制"按钮并列，提供两种输出方式

## 执行复盘

### 顺利的部分

1. **类型系统**：TypeScript 和 Rust 的类型检查帮助提前发现问题
2. **模块化设计**：复用现有的 `process_voice_input` 函数，减少重复代码
3. **UI 组件**：shadcn/ui 组件开箱即用，样式统一

### 遇到的问题

1. **文件读取方案**：
   - 最初尝试在前端使用 `@tauri-apps/plugin-fs` 读取文件
   - 发现该插件未安装，需要额外配置
   - 改为在后端创建新命令 `process_recording_file`，直接读取文件

2. **命令参数命名**：
   - Tauri 命令参数使用 snake_case（Rust 风格）
   - 前端调用时需要使用 camelCase（JavaScript 风格）
   - Tauri 会自动转换，但需要注意一致性

### 改进建议

1. **录音状态通知**：
   - 当前录音状态只在前端管理
   - 可以考虑使用 Tauri 事件系统，让后端主动通知前端录音状态变化

2. **录音文件清理**：
   - 当前录音文件保存在 app_data_dir/recordings/
   - 可以添加定期清理机制，避免占用过多磁盘空间

3. **录音时长限制**：
   - 当前没有录音时长限制
   - 可以添加最大录音时长（如 5 分钟），避免生成过大文件
