# XiLuoLin 产品需求分析

> 本文档用于明确项目定位、目标用户、核心场景、功能边界、使用流程和持续验收标准，作为产品设计、路线图、开发拆分和社区协作的基础。

## 1. 项目定位

XiLuoLin 是一个面向办公、写作和编程场景的开源 AI 语音输入助手，由个人发起并面向社区长期维护。

它帮助用户通过语音快速生成可直接发送、可直接使用的文本，减少打字、修改和润色成本。项目聚焦从语音采集到桌面输出的完整工作流，并持续提升可靠性、隐私、跨平台体验和可扩展性。

核心价值是：

- 把口头表达快速转换为清晰文本。
- 自动补标点、断句、去除口头禅。
- 通过不同“人格”生成适合不同场景的文本。
- 降低从“想到”到“输入完成”的操作成本。

## 2. 目标用户

### 2.1 开发者

开发者在使用 Agent 工具、编写 Prompt、描述需求或记录调试思路时，经常需要快速组织较长文本。语音输入可以减少反复打字和修改成本，让用户先自然说出想法，再由 AI 整理成结构化 Prompt 或任务描述。

### 2.2 管理者与协作者

管理者或团队协作者在发布任务、回复消息、同步需求时，需要把口头指令转成清晰、结构化、语气温和的文本。工具应帮助他们减少措辞成本，提高沟通效率。

### 2.3 创作者

创作者在突然想到灵感时，需要快速记录。工具应支持将语音内容整理成标题、要点、草稿或待办，方便后续保存、复用和继续创作。

## 3. 核心使用场景

### 3.1 Agent Prompt 输入

用户在使用 Agent 工具时，通过语音描述目标、上下文、约束和输出要求。系统将口述内容整理成更清晰的 Prompt，减少用户手动编辑成本。

示例输出方向：

- 结构化任务说明
- 清晰的目标和限制条件
- 更适合交给 AI Agent 执行的 Prompt

### 3.2 任务发布与消息回复

用户口述任务背景、要求和截止时间。系统将内容整理成结构明确、语气自然的任务说明或消息回复。

示例输出方向：

- 正式任务指令
- 温和消息回复
- 工作沟通文本

### 3.3 灵感记录与写作整理

用户在想到创意、文章主题、产品想法或待办事项时，直接用语音记录。系统将口述内容整理成更适合保存的文本。

示例输出方向：

- 标题候选
- 要点列表
- 草稿段落
- 待办清单

## 4. 功能范围

### 4.1 做什么

#### 语音输入

工具提供语音输入能力。用户说话后，系统快速生成文本结果。

#### 文本清理

系统自动补充标点、断句、去除口头禅和重复表达，让口述内容更像可直接发送或保存的文字。

#### 人格化改写

系统支持人格化输出。人格代表一组特定的表达风格、输出结构和适用场景。

系统提供默认人格和内置人格，例如：

- Prompt 工程师
- 任务协作者
- 灵感整理师
- 正式消息助手

用户也可以创建自定义人格，用于定义自己常用的表达方式、语气偏好和输出格式。用户在语音输入前或输入后可以选择人格，系统会根据当前人格将语音内容整理成对应风格的文本。

#### 热词词典

系统支持热词词典，用于提高专有名词、人名、项目名、技术词的识别和修正准确度。

#### 一键复制或粘贴

系统支持将生成结果一键复制，或粘贴到当前输入位置，降低从语音到实际输入的操作成本。

#### 历史记录

系统保存最近输入历史，方便用户回看、复用和二次编辑。

#### 数据统计卡片

系统基于历史记录生成轻量数据统计，用于让用户直观看到语音输入带来的效率收益。

优先展示：

- 语音协作次数
- 累计口述时间
- 口述生成字数
- 预计节省时间
- 常用人格

这些统计应优先在本地计算和保存，作为个人效率反馈，并帮助用户理解工具在真实工作流中的价值。

### 4.2 不做什么

当前产品边界暂不包含以下内容：

- 不做完整系统级输入法内核。
- 不做语音模型训练，优先接入现成 ASR 服务或开源模型。
- 不做会议纪要、长音频转写等重型语音处理场景。
- 不做复杂多人协作平台。
- 不做全功能文本编辑器，只提供轻量编辑和确认能力。

## 5. 预期使用流程

### 5.1 首次使用流程

1. 用户首次打开工具。
2. 用户配置 ASR 服务或本地语音识别模型。
3. 用户选择默认人格。
4. 用户配置用于人格调整的快速文本模型。
5. 用户配置可选快捷键、热词词典、粘贴方式和自定义人格。
6. 系统保存配置，进入可用状态。

### 5.2 日常输入流程

1. 用户在聊天软件、代码编辑器、文档或浏览器等需要输入文本的位置打开工具。
2. 用户按下快捷键，选择长按或免长按模式开始录音。
3. 用户自然说话。
4. 系统录音并在录音完成后调用 ASR 服务或本地语音识别模型。
5. ASR 返回原始识别文本。
6. 系统将原始识别文本、当前人格提示词、热词词典和必要上下文发送给快速文本模型。
7. 快速文本模型返回完成文风调整和结构整理后的文本。
8. 系统自动粘贴到当前输入位置，或提供一键复制。
9. 系统保存本次输入历史，方便用户回看和复用。
10. 系统更新本地统计数据，用于展示累计使用效果。

## 6. 核心能力拆解

### 6.1 录音能力

系统需要支持基础录音控制，包括开始录音、停止录音、录音状态提示和异常状态提示。

优先支持：

- 快捷键长按录音
- 快捷键按一下开始再按一下结束的免长按录音

### 6.2 语音识别能力

系统通过现成 ASR 服务或开源模型完成语音转文字。Provider 选择应综合考虑响应速度、稳定性、隐私、成本、可维护性和社区可扩展性。

输入是音频文件，输出是原始识别文本。

语音识别阶段只负责把音频转换成文本，不负责人格化改写。人格提示词不直接发送给语音识别模型。

### 6.3 AI 整理能力

系统根据用户选择的人格，将原始识别文本整理成可直接使用的文本。

AI 整理阶段由快速文本模型完成。系统将原始识别文本、人格提示词、热词词典和必要上下文组合成文本请求，让模型完成断句、补标点、去口头禅、语气调整和结构整理。

不同人格应有不同的整理目标：

| 人格 | 适用场景 | 整理目标 |
|---|---|---|
| Prompt 工程师 | Agent Prompt、编程辅助 | 明确目标、上下文、约束和输出格式 |
| 任务协作者 | 任务发布、需求沟通 | 补全背景、拆分要求、保持语气清晰 |
| 灵感整理师 | 写作、创作、想法记录 | 提炼标题、要点、待办或草稿 |
| 正式消息助手 | 办公消息、邮件、回复 | 保持礼貌、清楚、适合发送 |

### 6.4 人格系统能力

人格是文本整理风格的核心配置。每个人格定义一种输出方式，系统根据当前人格决定如何整理用户的语音内容。

人格至少包含：

| 字段 | 说明 |
|---|---|
| 名称 | 人格名称，例如 Prompt 工程师 |
| 描述 | 说明这个人格适合处理什么输入 |
| 适用场景 | 编程、办公、写作、消息回复等 |
| 输出语气 | 正式、温和、直接、结构化、创意化等 |
| 输出结构 | 段落、列表、标题加要点、Prompt 模板等 |
| 默认提示词 | 用于指导 AI 整理文本的系统提示词 |
| 是否默认 | 用户未手动选择时使用的人格 |

系统支持内置人格、默认人格和自定义人格，并持续完善人格的管理、导入导出和共享方式。

### 6.5 热词修正能力

用户可以维护常用热词，例如人名、项目名、技术栈、产品名和缩写。系统在识别和整理阶段优先保留这些词，减少误识别。

### 6.6 输出能力

系统需要支持两类输出方式：

- 一键复制到剪贴板。
- 自动粘贴到当前输入位置。

输出链路应同时提供一键复制、自动粘贴和失败降级，并在不同平台上明确能力与权限边界。

### 6.7 历史记录能力

系统保存最近输入历史，至少包含：

- 原始识别文本
- 整理后的结果
- 使用人格
- 口述时长
- 生成字数
- 创建时间

历史记录使用本地 SQLite 保存，并为数据迁移、备份和未来存储演进保留清晰边界。

### 6.8 数据统计能力

系统基于历史记录计算用户的语音输入使用情况和效率收益。统计数据用于帮助用户理解工具在真实工作流中的使用情况和效率收益。

当前版本支持并持续完善以下统计卡片：

| 指标 | 说明 | 计算方式 |
|---|---|---|
| 语音协作次数 | 用户完成语音输入并生成文本的次数 | 每次成功生成结果后加 1 |
| 累计口述时间 | 用户累计录音时长 | 汇总每条历史记录的口述时长 |
| 口述生成字数 | AI 整理后生成的文本总字数 | 汇总整理结果字数 |
| 预计节省时间 | 相比手动打字节省的时间估算 | 按平均打字速度估算输入耗时，再减去口述时长 |
| 常用人格 | 用户最常使用的人格 | 按历史记录中的人格使用次数排序 |

预计节省时间只作为估算值，用于展示趋势和价值。当前可使用固定估算规则，例如按每分钟手动输入 80 个中文字计算；后续应允许用户配置平均打字速度，使结果更贴近个人习惯。

## 7. 核心产品基线

项目的核心产品基线包含以下闭环：

1. 用户可以录音或上传一段短音频。
2. 系统可以调用 ASR 服务生成原始文本。
3. 系统可以调用快速文本模型，根据当前人格对原始文本进行整理。
4. 用户可以查看整理结果。
5. 用户可以复制结果。
6. 系统可以保存最近输入历史。
7. 系统可以展示基础统计卡片。

持续开发优先级：

| 优先级 | 功能 | 说明 |
|---|---|---|
| P0 | 录音 / 音频输入 | 没有语音输入就无法成立 |
| P0 | ASR 转写 | 形成语音到文本的基础能力 |
| P0 | 快速文本模型整理 | 根据人格完成文风调整和结构整理 |
| P0 | 复制结果 | 保证结果可直接使用 |
| P1 | 内置人格 | 提供 Prompt、任务、灵感、正式消息等系统人格 |
| P1 | 默认人格 | 用户可以选择默认输出人格 |
| P1 | 历史记录 | 支持回看和复用 |
| P1 | 数据统计卡片 | 展示协作次数、口述时间、生成字数和节省时间 |
| P1 | 热词词典 | 提高专业场景识别效果 |
| P2 | 自定义人格 | 用户可以创建和编辑自己的输出人格 |
| P2 | 自动粘贴 | 降低操作成本 |
| P2 | 全局快捷键 | 提升桌面使用效率 |

## 8. 非功能需求

### 8.1 易用性

用户应能在较少操作内完成从说话到得到可用文本的流程。核心路径应尽量短，避免复杂配置阻断首次体验。

全局快捷键录音开始时应记录目标应用。录音指示器和后续处理不得抢占键盘焦点；处理期间即使用户切换窗口，也应优先向录音开始时的目标窗口投递文本。

设置页应在用户开始录音前展示麦克风、ASR、文本 Provider、全局快捷键和自动粘贴能力。麦克风或模型配置缺失属于阻断项；自动粘贴不可用属于软限制，不得阻断识别、历史保存和复制结果。

### 8.2 响应速度

短语音输入应尽快返回结果。MVP 阶段可接受数秒等待，但需要明确区分录音、识别、整理、输入、完成和失败状态。
短语音输入应尽快返回结果。即使外部 Provider 需要数秒响应，也必须明确展示处理中状态，并支持超时、失败提示和可恢复操作。

### 8.3 可解释性

用户应能理解系统做了什么：当前是在录音、识别、整理还是已完成。

### 8.4 可复现性

项目需要提供清晰的运行方式、依赖说明、验证步骤和兼容性边界，确保用户与贡献者可以复现核心功能。

### 8.5 隐私与数据边界

如果音频会发送给第三方 ASR 服务，或文本会发送给第三方快速文本模型，需要在 README 或设置页中说明。API Key 应保存在操作系统凭据库中，不得明文写入项目文件或 Tauri Store；旧版明文配置只有在安全迁移成功后才能清理。

历史记录和统计数据默认保存在本地。统计卡片只展示个人使用数据，不应默认上传到服务器或用于多人排行。历史记录应保存输入来源、实际 Provider/模型、是否发生降级和投递方式等非敏感快照。

应用生成的录音默认视为临时文件，只允许在应用录音目录中处理，并应在处理成功或失败后清理。只有用户显式开启保留、自动保存历史开启且历史写入成功时才能保留；删除历史时同步删除关联录音。已保留录音应支持试听和使用当前 Provider 重新转写；即使没有录音，历史原始文本也应支持使用当前人格重新整理。用户主动选择的外部音频不得被临时文件清理逻辑删除。

运行日志不得输出 API Key 片段、用户原始或整理后文本、完整录音路径等敏感内容。

自动粘贴不得永久覆盖用户原剪贴板。成功粘贴后应尽力恢复原文本或图片剪贴板；目标窗口恢复或粘贴失败时，生成文本应保留在剪贴板中作为手动粘贴兜底。

## 9. 验收标准

每个稳定版本都应满足以下基础标准：

- 用户可以完成一次完整的语音输入流程。
- 系统可以将短语音转换成文本。
- 系统可以对口语化文本进行断句、补标点和轻度润色。
- 系统可以按至少 3 个内置人格输出不同风格的文本。
- 用户可以选择默认人格。
- 用户可以复制生成结果。
- 系统可以保存最近输入历史。
- 系统可以展示语音协作次数、累计口述时间、口述生成字数和预计节省时间。
- README 中说明项目定位、运行方式、依赖、使用方式、隐私边界、路线图和贡献入口。

## 10. 持续迭代方向

后续可以逐步扩展：

- 支持自动粘贴到当前输入位置。
- 支持全局快捷键和悬浮录音状态。
- 支持更多 ASR Provider。
- 支持本地离线 ASR。
- 支持自定义人格。
- 支持人格导入导出。
- 支持历史搜索和收藏。
- 支持热词导入导出。
- 支持按人格、日期、应用场景查看统计趋势。
- 支持用户自定义平均打字速度，用于更准确估算节省时间。

## 11. 需求结论

本项目的关键不是追求完整系统输入法内核，而是持续建设一个可靠、可解释、可复现、可扩展的开源 AI 语音输入助手。核心链路是“语音输入 → ASR 转写 → 文本模型按人格整理 → 复制或粘贴 → 历史保存”，后续工作围绕可靠性、隐私、跨平台体验、Provider 扩展和社区协作逐步演进。
