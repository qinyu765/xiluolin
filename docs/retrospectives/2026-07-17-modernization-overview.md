# 2026-07-17 XiLuoLin 现代化改造总览

## 1. 文档目的

本文记录 2026 年 7 月 16 日至 7 月 17 日对 XiLuoLin 进行的一轮集中现代化改造，包括：

- 为什么项目需要重新规划；
- 如何从现有代码和文档识别真实问题；
- 为什么参考 Voicebox，但没有照搬其完整产品范围；
- M0–M3 每个阶段做了什么；
- 实施过程中出现了哪些工程、环境和协作问题；
- 最终如何验证和落地；
- 哪些内容仍需继续完成。

本文不是 API 手册，而是一份可供维护者和后续 Agent 回顾决策过程的工程复盘。

## 2. 起点：一个“功能很多，但基线不可靠”的 MVP

最初检查仓库时，项目已经具备较完整的功能列表：

- Tauri + React 桌面应用；
- 麦克风录音和全局快捷键；
- 智谱 ASR 与 OpenAI-compatible 文本整理；
- 人格、热词、历史记录和统计；
- 剪贴板和模拟粘贴；
- 录音状态窗口。

但“代码中存在功能”不等于“产品链路可靠”。检查暴露了几个基础问题：

1. `pnpm build` 可以成功，但 `pnpm exec tsc --noEmit` 有 6 个错误。
2. 构建脚本只调用 Vite，没有把类型检查作为构建前置条件。
3. 当前机器最初没有 Rust 工具链，仓库也没有可直接观察的完整质量门禁。
4. API Key 明文保存在 Tauri Store，并有日志输出密钥片段。
5. 录音文件处理后可能长期残留。
6. 自动粘贴只面向“处理完成时的当前窗口”，没有锁定录音开始时的目标。
7. 状态窗显示时调用 `set_focus()`，可能主动破坏用户原来的输入焦点。
8. 自动粘贴会覆盖用户剪贴板，而且不会恢复。
9. README、任务跟踪和实际代码状态存在矛盾。
10. `main.tsx`、`data.rs` 等文件较大，前后端配置类型已经发生漂移。

因此下一步不应该立即堆叠更多模型或页面，而应先修复工程基线和核心输入链路。

## 3. 路线调整：参考 Voicebox，但保持产品聚焦

Voicebox 提供了一个有价值的参考：它没有把语音输入理解为“录音后调用一次 ASR”，而是把 Dictation 设计为完整闭环：

```text
快捷键开始
  → 保存目标应用
  → 录音状态反馈
  → 转写
  → 整理
  → 恢复目标窗口
  → 粘贴
  → 保存 Capture
```

值得吸收的部分：

- 录音开始时保存焦点，而不是处理结束时猜测目标；
- 状态悬浮窗预创建并且不抢焦点；
- 原始文本和整理结果可追溯；
- 自动粘贴失败时仍保留结果；
- 本地模型是明确的隐私能力；
- Capture 可以重新转写和重新整理。

明确不照搬的部分：

- TTS 与声音克隆；
- MCP 语音输出；
- 长音频、故事编辑和完整 voice studio；
- Python sidecar 与多种重型推理后端；
- 复杂的 agent orchestration。

最终产品路线确定为：

> 轻量 AI 语音输入助手，云端优先、可选本地；优先保证跨应用输入可靠性，再增加可追溯历史和离线能力。

## 4. 调整后的阶段划分

### M0：稳定与安全基线

目标：证明仓库可以稳定编译、测试，并消除明显的凭据和录音安全风险。

对应主要任务：

- T029：建立前后端质量检查基线；
- T030：加固凭据和隐私数据处理。

### M1：跨应用输入可靠性

目标：让语音结果稳定回到用户开始说话时的输入位置，而不是依赖处理结束时的焦点。

对应任务：

- T031：CaptureSession 与可靠文本投递；
- T032：输入就绪检查。

### M2：可追溯 Capture

目标：让历史记录从静态文本列表升级为可解释、可复用的 Capture。

对应任务：

- T033：历史快照与可选录音保留；
- T034：试听、重新转写和重新整理。

### M3：本地 ASR 与混合 Provider

目标：提供离线转写，同时确保本地模式不会静默上传云端。

对应任务：

- T035：本地 Whisper ASR 与显式云端降级。

## 6. 实施过程中的关键转折

### 6.1 先发现“构建成功”是假象

第一次执行：

```bash
pnpm build
```

Vite 构建成功，但单独执行：

```bash
pnpm exec tsc --noEmit
```

发现多个类型错误，包括：

- 已删除的 `output_mode` 仍被旧设置弹窗引用；
- 动态 Lucide 图标类型不安全；
- 快捷键组件调用不存在的 setter。

这说明构建命令不能只检查产物生成，还必须检查类型和测试。

### 6.2 从“清理日志”升级到“重新设计凭据持久化”

最初问题看起来只是日志输出了 API Key 前缀，但进一步检查发现完整 API Key 会跟随 `AppConfig` 写入 `settings.json`。

因此最终方案不是单纯删除日志，而是拆分持久化：

```text
非敏感配置 → Tauri Store
API Key     → 系统凭据库
业务数据     → SQLite
```

并加入旧明文配置迁移，遵循“先安全写入，后清除明文”。

### 6.3 从“粘贴文本”升级到 CaptureSession

原输出逻辑只做：

```text
写入剪贴板 → 等待 20ms → 模拟 Ctrl/Cmd + V
```

它不知道用户是在哪个窗口开始录音的，而且状态窗还会抢焦点。修复不能只增加延迟，而需要在录音开始时建立会话上下文。

由此产生 CaptureSession 状态机和 `session_id`，成为后续历史关联、投递方式更新和失败恢复的基础。

### 6.4 从“始终删除录音”调整为“默认删除、显式保留”

纯隐私方案倾向始终删除录音，但 Capture 重新转写需要原始音频。

最终策略：

- 默认删除；
- 用户显式开启；
- 自动保存历史开启；
- 历史写入成功；
- 以上全部成立才保留。

这避免为了后续功能牺牲默认隐私边界。

### 6.5 本地 ASR 不只是增加一个函数

本地 ASR 实际包含：

- 当前 crate/API 重新调研；
- CMake 和 whisper.cpp 构建；
- 模型下载、临时文件、进度和校验；
- WAV 解码和 16 kHz 重采样；
- 模型上下文缓存；
- whisper.cpp 敏感日志抑制；
- 本地失败后的显式云端降级；
- 历史记录实际 Provider；
- 真实模型 smoke test。

这说明“接入本地模型”是一个完整子系统，而不是一个 Provider 分支。

- 构建不再绕过类型检查；
- 前后端质量检查统一；
- Windows CI 可验证 Tauri/Rust；
- 敏感配置和业务数据职责分离；
- 新增数据库迁移和安全文件操作测试。

### 产品层

- 跨应用语音输入有明确会话和状态；
- 自动粘贴不会永久污染剪贴板；
- 历史记录可以解释结果来源；
- 原始录音可按需保留和复用；
- 本地 ASR 默认离线，不静默上传。

### 协作层

- 项目转向长期维护的开源定位；
- 建立双语 README、贡献指南、安全策略和路线图；
- Agent 规则统一；
- 提交发布不再要求每次人工批准；
- 并发任务使用 worktree 隔离。

## 10. 核心经验

1. **先证明基线可靠，再增加功能。**
2. **桌面语音输入的核心是焦点和失败恢复，不只是 ASR 准确率。**
3. **隐私默认值必须体现在代码流程，而不只是文案。**
4. **可切换 Provider 必须共享同一配置选择事实源。**
5. **需要后续更新的数据不能依赖 fire-and-forget 写入。**
6. **真实模型测试不能完全被 mock 测试替代。**
7. **多 Agent 并发必须隔离工作树，否则很容易覆盖或混合提交。**
8. **PR、分支和 main 的状态必须最终对账。**
