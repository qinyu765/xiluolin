## 任务目标

实现文本输出到当前光标位置，主链路使用键盘注入，剪贴板作为兜底方案。

## 实际改动

### 架构调整

参考 `类似语音输入产品实现方案分析.md` 和 `OpenTypeless` 实现，采用"直接键盘注入 + 剪贴板兜底"架构，而非"先复制再粘贴"：

1. **主链路**：直接键盘注入（`keyboard_inject`）- 将文本逐字符注入到当前焦点窗口
2. **降级链路**：剪贴板粘贴（`clipboard_paste`）- 写入剪贴板后模拟 Ctrl+V / Cmd+V
3. **最终兜底**：仅复制（`clipboard_copy`）- 至少保证文本在剪贴板中，提示用户手动粘贴

### 实现文件

**`src-tauri/src/output.rs`**（136 行）

- `OutputMethod` 枚举：`Keyboard`、`Clipboard`、`Manual` 三种输出方式
- `OutputResult` 结构：返回实际使用的方法、成功状态和用户提示消息
- `output_text` 命令：自动选择最佳输出方式的主入口
- `keyboard_inject`：使用 `enigo` 逐字符输入，处理换行、Tab 等特殊字符
- `clipboard_paste`：先复制到剪贴板，再模拟粘贴快捷键（跨平台支持 macOS Cmd+V 和 Windows/Linux Ctrl+V）
- `clipboard_copy`：使用 `arboard` 写入剪贴板

- `enigo = "0.2"`：键盘模拟
- `arboard = "3.4"`：剪贴板访问

## 为什么这么做

### 1. 为什么主链路是键盘注入而非剪贴板

参考文档明确指出：

- 用户真正要的是"文字已经在输入框里"，而非"文字在剪贴板里等待粘贴"
- 剪贴板是全局共享资源，污染成本高（覆盖用户刚复制的内容）
- 直接注入更符合"无感输入"的产品体验，不需要额外手动粘贴

### 2. 为什么需要三层降级

- **键盘注入**：体验最好，但可能因权限、焦点、受保护输入框失败
- **剪贴板粘贴**：兼容性更好，但仍可能因权限或应用拦截失败
- **仅复制**：最终兜底，至少保证文本不丢失

### 3. 为什么逐字符输入而非批量

`enigo` 的 `text()` 方法在某些平台和应用中对中文、特殊字符支持不稳定，逐字符输入更可靠。添加 500 微秒延迟避免输入过快导致丢字。
